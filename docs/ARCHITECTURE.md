# Архитектура Meaning («Смысл»)

## Инвариант

Meaning («Смысл») использует проверяемую гибридную архитектуру, потому что небольшая LLM не
должна быть источником предметной истины:

1. правильность закрытого или структурированного задания определяет предметный
   модуль по ключу и атомарным компонентам правила;
2. предметный модуль сохраняет факты ответа, балл каждого компонента и точный диагноз;
3. локальная LLM получает только этот проверенный контракт, объясняет ошибку,
   проводит проверку понимания и оценивает открытый ответ;
4. код калибрует LLM-оценку по покрытию обязательных понятий;
5. LLM создаёт восстановительную практику и предложение следующего занятия;
6. BKT обновляет накопленное освоение по всем свидетельствам цикла.

Практические миссии образуют соседний, изолированный контур. Они не меняют предметный модуль
или таблицу обычных попыток: сценарий, диалог и трассы лежат в `mission_attempts`, а
mastery обновляется только после достижения коммуникативной цели.

Голосовой диалог — ещё один изолированный контур. Он не меняет баллы исходного
задания: `voice_sessions` и `voice_turns` хранят собственные реплики и speaking-
метрики, а преподаватель видит их отдельным evidence-слоем.

Для открытого письма, где полного ключа нет, работает LLM-рубрика. Для проверки понимания нет
эвристического ответа вместо модели: недоступность GGUF или llama.cpp останавливает
проверку с явной ошибкой. JSON — только контракт передачи данных, а не способ
определить правильность.

## Локальные модели

| Компонент | Значение |
|---|---|
| Быстрый интерактивный контур | Qwen2.5-3B-Instruct Q4_K_M, около 2,1 ГБ |
| Углублённый режим | Qwen2.5-7B-Instruct Q4_K_M, около 4,7 ГБ |
| Квантизация | Q4_K_M |
| Формат | GGUF |
| Runtime | llama.cpp |
| Интерфейс | локальный OpenAI-compatible HTTP на 127.0.0.1 |
| Интернет при inference | не нужен |
| API-ключ | не нужен |

### Локальный голосовой контур

| Компонент | Ответственность |
|---|---|
| Browser Audio API | захват микрофона, 16 kHz PCM, energy gate и barge-in |
| WebSocket | постоянный двунаправленный транспорт аудио и событий |
| Silero VAD 6.2 | фильтрация речевых сегментов |
| whisper.cpp + base.en | локальный английский ASR |
| LanguageTool 6.6 | независимые grammar rules |
| Предметные правила Meaning | инварианты, которых нет в общем средстве проверки |
| Qwen 3B / 7B | контекстный диалог, смысл и grounded feedback |
| Windows SAPI | локальный TTS |

Для relative clauses код отдельно извлекает `antecedent`, `who/which/where` и
defining/non-defining сигнал. Qwen получает этот evidence вместе с `last_tutor_turn`
и списком уже заданных вопросов. Повтор упражнения, противоречие структурной находке
или ложное «Your example is correct» заменяется проверяемой grounded-репликой.

Это полнодуплексный транспорт с barge-in: микрофон слушает во время TTS, новая речь
останавливает воспроизведение и инвалидирует старую генерацию. При этом ASR в MVP
остаётся turn-segmented — Whisper запускается после 700 мс тишины, а не транслирует
частичные слова.

### Рекомендуемый Realtime-контур

| Компонент | Ответственность |
|---|---|
| Browser WebRTC | непрерывный микрофон и воспроизведение speech-to-speech без PCM-прокси |
| Серверный `/realtime/calls` proxy | создаёт сессию стандартным API-ключом; ключ не попадает в браузер |
| `gpt-realtime-2.1` | понимание входного аудио и естественный голосовой ответ |
| far-field noise reduction | фильтрация ноутбучного/комнатного микрофона до VAD и модели |
| server VAD 5,2 с | паузы до пяти секунд не дробят мысль; `interrupt_response` реализует barge-in |
| `gpt-4o-transcribe` | асинхронный транскрипт для UI и evidence, не источник восприятия модели |
| `gpt-5.6-luna` | фоновая grounded-оценка в строгом JSON Schema |
| SQLite | реплики, оценки, модели и trace/call ID для кабинета преподавателя |

Оценщик запускается после голосового ответа в отдельном HTTP-потоке и не блокирует
следующую реплику. Если оценщик временно недоступен, разговор продолжается, транскрипт
помечается как неоценённый, а нулевой балл не выдаётся пользователю как содержательный.

`scripts/setup_local_llm.ps1` закрепляет версии и проверяет SHA-256. `src/meaning_trainer/local_llm.py` автоматически поднимает локальные серверы и принимает только структурированный JSON. По умолчанию весь синхронный путь использует 3B-модель; `LOCAL_LLM_QUALITY_MODE=1` переводит сложную проверку и финализацию на 7B.

## Состояния попытки

```text
DRAFT
  ↓ отправка ответа
SUBJECT_GRADER / LLM_OPEN_TASK_RUBRIC
  ├── is_correct=true  → VERIFIED_KNOWLEDGE_CHECK_CONTRACT
  └── is_correct=false → VERIFIED_DIAGNOSTIC_CONTRACT
                          ↓
               1 LLM-вызов → 2 вопроса
                          ↓
                 LLM_EVIDENCE
                          ↓
                      BKT_UPDATE
                          ↓
               LLM_ADAPTIVE_ACTION
                  ├── TRANSFER
                  └── REMEDIATION
                          ↓
             LLM_COHORT_RECOMMENDATION
                          ↓
                        SAVED
```

## Состояния практической миссии

```text
MISSION_BRIEF
  ↓
NPC_OPENING → STUDENT_TURN
  ↓                 ↓
детектор 2 обязательных форм + 1 локальный LLM-вызов
  ↓
фильтр: точная цитата → локальная правка → предметный guardrail
  ├── ACTIVE (до 3 ходов)
  ├── NEEDS_RETRY
  └── COMPLETED → mastery update
```

Валидатор не принимает уверенность модели за факт. Итог объединяет покрытие
обязательных форм (45%), грамматическую оценку (30%) и достижение коммуникативной
цели (25%). Без всех обязательных форм завершить миссию нельзя. Ошибка LLM
показывается лишь тогда, когда её фрагмент буквально присутствует в текущей
реплике и исправление остаётся локальной правкой, а не новым ответом.

## Состояния голосового диалога

```text
MIC_ACTIVE ⇄ BOT_AUDIO
    ↓ speech detected during playback
BARGE_IN → cancel generation id → SPEECH_BUFFER
    ↓ 700 ms silence
WHISPER + SILERO → TRANSCRIPT
    ↓
LANGUAGETOOL + STRUCTURAL RULES ║ QWEN DIALOGUE
    ↓ grounded merge
SCORES + CORRECTION + NEXT QUESTION
    ├── WebSocket text/audio → browser
    └── voice_turns → teacher pulse
```

В рекомендуемом режиме этот state machine заменяется нативным WebRTC-контуром:

```text
MIC ⇄ WEBRTC ⇄ GPT-REALTIME-2.1 AUDIO
        ├── server VAD 5,2 с + interrupt_response
        ├── gpt-4o-transcribe → captions/evidence
        └── stable turn ID → ordered gpt-5.6-luna JSON Schema → voice_turns
```

Служебные реплики к собеседнику не оцениваются. Каждой содержательной реплике
назначается стабильный `client_turn_id`; её транскрипт, ответ и карточка оценки
остаются связанными даже при задержке транскрипции или фоновой проверки.

Qwen не является единственным судьёй грамматики. Числовые поля ограничиваются
кодом, проверенная структурная находка имеет приоритет над выводом модели, а
сомнительный LLM-only случай можно перепроверить локальной 7B. Темп, доля пауз и
fillers считаются по PCM и транскрипту; произношение намеренно не оценивается.

## Разделение двух оценок

`Evidence.score` — качество конкретного ответа на конкретный вопрос сейчас.

`mastery[skill_id]` — накопленная вероятностная оценка BKT с учётом истории. Она может оставаться выше нуля после плохого ответа и не должна называться «точностью ответа».

Интерфейс показывает оба значения рядом, но с разными подписями. Выбор коррекционного задания опирается на полный LLM-анализ текущей попытки, а не только на старое mastery.

## Структурированные LLM-контракты

### Проверка задания

- `submission_score`;
- `is_correct`;
- `feedback`;
- `mode`;
- `skill_results`;
- `criterion_results` с точной ошибкой и исправлением;

Для структурированных заданий эти поля строит предметный grader. Например,
`Relative Clauses` отдельно проверяет сохранение исходных фактов, выбор relative
word, antecedent, defining/non-defining смысл и пунктуацию. Поэтому допустимая
перестройка предложения принимается, а частичная ошибка не превращается в 0%.

Grader формирует контракт первого вопроса по конкретному компоненту. Один короткий
LLM-вызов предлагает сразу формулировку проверки понимания и перенос того же компонента в новый
контекст; в строгом режиме первый вопрос берётся из проверенного контракта, а
LLM-вариант не может незаметно заменить проверяемое правило другим. Семантическую
оценку обоих свободных ответов выполняет локальная LLM.

### Проверка ответа

- `verdict`, `score`, `confidence`;
- что верно и что нужно улучшить;
- корректный ответ и обработка опечатки.

Промпт отдельно фиксирует: совпадение отдельных слов, повтор вопроса, «не знаю» и случайный текст должны получать 0–0.10.

### Финализация

- `branch` (`transfer` или `remediation`);
- персональная активность студента;
- тема, обоснование и план занятия преподавателя.

Для remediation активность обязательно содержит отдельные `explanation`, `worked_example` и `practice_task`; промпт требует перепроверить их по рубрике и типичным ошибкам.

Чтобы малая модель не могла исказить базовое правило, восстановительная ветка использует защищённый поиск по базе правил: LLM определяет пробел и `rule_id`, а объяснение, примеры и ссылка берутся из версионируемой карточки `english_b2_rules.json`. Ветка выводится только из оценок задания и проверки понимания, а не из отдельных ключевых слов ответа.

## Аудит

Для каждого вызова сохраняются:

- `trace_id` ответа llama.cpp;
- имя модели;
- SHA-256 GGUF;
- этап пайплайна;
- UTC-время;
- длительность;
- prompt/completion tokens, если runtime их вернул.

Аудит показывается в обоих кабинетах. Это техническое доказательство, но не гарантия педагогической корректности — её проверяют eval-набором и экспертной разметкой.

## Данные

SQLite содержит students, assignments, attempts, mission_attempts, voice_sessions,
voice_turns и mastery. В attempts дополнительно хранятся оценка задания, ветка,
персональное действие, групповая рекомендация и JSON-трассы. `mission_attempts`
хранит состояние и диалог отдельно; `voice_turns` — транскрипт, ответ, акустические
метрики, независимые grammar findings, speaking-оценку и LLM trace.

Задания английского B2 лежат в `data/english_b2_assignments.json`, миссии — в `data/english_b2_missions.json`, проверенные карточки правил и ссылки — в `data/english_b2_rules.json`, граф навыков — в `data/curriculum.json`.

## Риски

| Риск | Мера |
|---|---|
| Небольшая LLM ошибается | предметный grader, атомарный контракт, structured output, eval-набор, показ rationale, апелляция преподавателю |
| Модель засчитывает keyword matching | негативные тесты с бессмыслицей и перефразированием вопроса |
| Подмена LLM эвристикой | отсутствие fallback, trace ID и SHA-256 в каждой попытке |
| Медленная работа на CPU | 3B быстрый контур, постоянный сервер, короткие JSON-ответы; 7B включается отдельно на мощном ПК |
| Утечка постоянного API-ключа | ключ хранится только сервером; браузер получает SDP-канал через локальный proxy |
| Передача голосовых данных в облако | явный выбор Realtime; рядом сохранён полностью локальный fallback |
| Смешение mastery и текущего score | разные поля, подписи и тесты |
| Галлюцинация ошибки в миссии | буквальная цитата, проверка локальности исправления, предметный guardrail |
| Низкая уверенность проверки понимания | явная отметка и очередь ручного просмотра в кабинете преподавателя |
| ASR исказил слово | отдельный транскрипт в UI, запрет выдавать оценку произношения, минимальная коррекция |
| LLM пропустила устную ошибку | LanguageTool + предметный структурный валидатор до объединения результата |
| Бот повторяет один вопрос | история `prior_tutor_turns`, similarity guard и персонализированный follow-up по antecedent |
| Эхо при barge-in | browser echo cancellation и повышенный порог речи во время TTS |
