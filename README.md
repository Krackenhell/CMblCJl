# VivaTrace

**VivaTrace — адаптивная проверка понимания для неоднородной учебной группы.**

Студенты одной университетской группы учатся по общей программе, но приходят с разным уровнем подготовки. Обычная LMS сохраняет оценки, однако не показывает, понимает ли студент собственное решение и что преподавателю стоит изменить на следующем занятии.

VivaTrace замыкает этот цикл:

1. преподаватель создаёт или изменяет задание и выбирает проверяемые навыки;
2. студент отправляет код или развёрнутый ответ;
3. анализатор находит в работе сигналы возможных ошибок;
4. система формирует персональную мини-защиту и при слабом ответе задаёт уточняющий вопрос;
5. ответы превращаются в проверяемые свидетельства, обновляющие BKT-модель освоения навыков;
6. последняя попытка каждого студента попадает в пульс группы;
7. преподаватель видит общие пробелы, персональные траектории и рекомендацию для следующей пары.

Это не детектор списывания и не автоматический экзаменатор. MVP проверяет способность объяснить предметное решение и оставляет высокорисковые решения преподавателю.

## Что реально работает

- три раздельных аккаунта студентов: Анна, Максим и Дарья;
- хранение заданий, попыток, ответов и состояния навыков в SQLite;
- изменение решения действительно меняет обнаруженные сигналы и подбор вопросов;
- вопросы привязаны к навыкам задания, уровню студента и содержанию работы;
- слабый ответ вызывает дополнительный уточняющий вопрос;
- преподаватель может создавать и редактировать задания;
- пульс группы пуст до прохождения защит и пересчитывается по реальным данным;
- в агрегате учитывается только последняя попытка каждого студента;
- локальная рубрика работает без API-ключа, а LLM-оценка подключается опционально.

В продукте оставлены только два интерфейса: кабинет студента и кабинет преподавателя. Архитектура, эксперимент и доказательство ценности находятся в документации и презентации, а не внутри пользовательского интерфейса.

## Быстрый запуск

```bash
python -m venv .venv
.venv\Scripts\activate
python -m pip install -r requirements.txt
streamlit run app.py
```

Приложение откроется на `http://localhost:8501`. При первом запуске автоматически создаются база `vivatrace.db`, три аккаунта и первое задание. Результаты студентов заранее не заполняются.

Для чистого повторения демонстрации преподаватель может открыть «Управление заданиями» и подтвердить очистку результатов.

## Опциональная LLM-оценка

Без ключа система использует воспроизводимую локальную предметную рубрику. Для LLM-режима:

```bash
copy .env.example .env
# задайте LLM_API_KEY, LLM_BASE_URL и LLM_MODEL
```

Переменные среды необходимо экспортировать перед запуском приложения. Если клиент или API недоступен, система автоматически возвращается к локальной рубрике.

## Сценарий демонстрации

1. В кабинете преподавателя показать пустой пульс: `0 из 3`.
2. Переключиться на Анну, изменить код, пройти персональную защиту.
3. Вернуться в пульс и показать первую строку, тепловую карту и рекомендацию.
4. Пройти защиту Максимом с более сильным решением, а Дарьей — со слабым объяснением.
5. Показать, что три разных решения дали разные ошибки, вопросы и уровни навыков.
6. В пульсе группы открыть повторяющиеся заблуждения и конкретные ответы студента.
7. Изменить условие задания в кабинете преподавателя и показать, что новая версия доступна студентам.

## Архитектура

```text
Работа студента
    ↓
Анализатор артефакта → гипотезы о пробелах
    ↓
Планировщик защиты → 3–4 вопроса по программе
    ↓
Оценщик свидетельств → балл + уверенность + точная цитата
    ↓
BKT-модель знаний → освоение каждого навыка
    ↓
Маршрутизатор → восстановление / практика / усложнение / проверка человеком
    ↓
SQLite → последняя попытка каждого студента
    ↓
Пульс группы → общие ошибки + действие для преподавателя
```

Подробнее: [архитектура](docs/ARCHITECTURE.md), [сценарий защиты](docs/PITCH.md), [переиспользование и атрибуция](docs/REUSE_AND_ATTRIBUTION.md).

## Data Science

`scripts/run_experiment.py` создаёт контролируемый синтетический набор и сравнивает два классификатора скрытого понимания студента:

- baseline: только оценка работы;
- hybrid: работа + ответы на мини-защите + сигналы анализатора.

Метрики: balanced accuracy, F1 и ROC-AUC. Это техническая валидация пайплайна, а не доказательство образовательного эффекта. Для такого вывода нужен пилот с реальными студентами и экспертной разметкой.

## Инженерное качество

- модульное типизированное ядро в `src/vivatrace`;
- персистентная SQLite-модель данных;
- тесты анализатора, BKT, рубрики, маршрутизации и хранилища;
- Docker и GitHub Actions;
- структурированный LLM-ответ и локальный запасной режим;
- проверка человеком при низкой уверенности;
- явная документация источников и ограничений.

Проверки:

```bash
pytest -q
ruff check .
python scripts/run_experiment.py
```

## Ограничения MVP

- предметная онтология и анализатор кода пока охватывают одну тему курса «Введение в ML»;
- выбор трёх аккаунтов сделан для демонстрации, промышленная авторизация не реализована;
- локальная рубрика ищет смысловые маркеры и слабее полноценной семантической модели;
- голосовой интерфейс и интеграция с LMS оставлены для следующей итерации;
- образовательный эффект требует реального пилота, которого синтетический эксперимент не заменяет.
